Sistema de Monitoramento de Estufa

(Especificação de Requisitos)

A - VISÃO GERAL DO SISTEMA

O sistema para Monitoramento de Estufa Greeno consiste do gerenciamento de dispositivos/ações relativas à uma estufa, tais como o controle da temperatura do ambiente, a irrigação dos cultivos, controle da iluminação e também da umidade ambiente.

O sistema deverá permitir que sejam agendadas diretivas personalizadas pelo usuário aos dispositivos, como por exemplo em que horários deve-se acender e/ou apagar as luzes.

O sistema será responsável por guardar os dados recebidos de diferentes sensores de ambiente: Sensor de Temperatura, de Umidade, de Luminosidade, de Energia, e outros Sensores que possam ser instalados. O sistema deve ser desenvolvido de uma forma que seja possível a instalação de novos tipos de sensores sem o impactar nas funcionalidades.

O sistema também será responsável por exibir diversos relatórios e consultas dos dados de controle armazenados em uma API.

B - REQUISITOS FUNCIONAIS
B1 - Lançamentos diversos

  1. O sistema deve permitir a inclusão, alteração e remoção de sensores de ambiente com os seguintes atributos: id do sensor (idSensor), nome do sensor (nomeSensor), unidade de medida (unidadeMedida), descrição da localização do sensor (descLocalizacao).

  2. O sistema deve permitir o armazenamento dos valores lidos dos sensores de ambiente no banco de dados de forma periódica, a qual deve ser estabelecida pelo usuário.

  3. O sistema deve permitir o cálculo das médias de medições por dia, semana e mês de leituras.

  4. O sistema deve permitir que seja definido um valor de medição limite para determinado sensor. O sistema deve emitir um aviso ao usuário quando este limite for atingido.

  5. O sistema deverá ter a implementação de logging em pontos críticos do sistema com diferentes níveis de Log.

B2 - Impressão de diversos tipos de relatórios e consultas

  1. O Sistema deverá disponibilizar a exibição para o usuário dos relatórios de um ou todos os sensores, separados por localidade, das seguintes maneiras:

    • Valor atual (Última Leitura)

    • Média do dia atual

    • Média dos últimos 7 dias

    • Média Mensal do mês atual das leituras diárias armazenadas

    • Gráficos de linha descrevendo as mudanças de consumo por mês, ou por semana

C - REQUISITOS NÃO FUNCIONAIS
C1 - Serviços

  1. O sistema deverá ser implementado em dois projetos distintos, um exclusivo para o backend (API e Consumidor de um Servidor de Mensagens) e outro exclusivo do frontend (Aplicação Web). Sendo estes implementados em Java\Spring e React respectivamente.

  2. O sistema de filas usado para a comunicação dos sensores para o backend utiliza o protocolo MQTT.

  3. O sistema deverá armazenar suas informações utilizando o Banco de Dados não relacional MongoDB.

  4. O sistema deve permitir um valor configurável para a frequência de escrita das leituras recebidas por mensagens no BD.

C1 - Confiabilidade

  1. O sistema deve ter capacidade para recuperar os dados perdidos da última operação que realizou em caso de falha.

  2. O sistema deve fornecer facilidades para a realização de backups dos arquivos do sistema.

  3. O sistema deve poder verificar que o banco seja acessado pelo front-end por token.

C2 - Eficiência

  1. O sistema deve responder a consultas on-line em menos de 5 segundos.

  2. O sistema deve iniciar a exibição de relatórios solicitados dentro de no máximo 20 segundos após sua requisição.

C3. Portabilidade

  1. O sistema deve ser executado em computadores com processadores compatíveis com Java e Node.js.

  2. O sistema deve ser capaz de armazenar os dados numa base de dados MongoDB.
